## 二、问题总结

### 2.1 核心问题概览

| 问题编号 | 问题描述 | 严重程度 | 影响范围 |
|---------|---------|---------|---------|
| P1 | 第三阶段仅处理B/C/D/OL级，SA/A级被排除在外 | 严重 | 高等级卖场无法获得补充分配 |
| P2 | 第三阶段分配上限过高（直接分配到15件），不考虑实际销量需求 | 严重 | B级卖场获得过多分配 |
| P3 | 销尽率计算未结合等级权重，低等级高销尽率卖场抢占资源 | 中等 | 分配优先级错乱 |
| P4 | 第二阶段销量匹配逻辑存在缺陷，未考虑等级差异 | 中等 | 分配不合理 |
| P5 | 第四阶段剩余分配无等级区分，按简单顺序分配 | 低 | 剩余资源分配不公 |

### 2.2 问题现象

从实际分配结果来看：
- B级卖场（如 PR呼和浩特特万百货）销尽率仅为 **17%**，但每个尺码分配了 **12件**
- A级卖场销尽率高达 **36%-63%**，却只分配了 **1-2件**
- 这与预期的"SA > A > B > C > D > OL"等级优先原则严重不符

## 三、问题详细分析与解决方案

### 3.1 问题一：SA/A级卖场在第三阶段被排除

#### 问题定位

**代码位置：** [allocation_core.py:130-154](file:///workspace/allocation_core.py#L130-L154)

```python
# 3. B/C/D/OL级按销尽率降序
bc_stores_sorted = []
for level in ['B', 'C', 'D', 'OL']:  # ← 问题：SA/A级被排除
    level_stores = [(store, store_data[store]['sell_through']) 
                   for store in stores_sorted 
                   if store_data[store]['level'] == level]
```

#### 问题分析

当前代码在第三阶段只处理 B/C/D/OL 级卖场，SA/A 级卖场完全被排除在外。这导致：

1. SA/A级卖场在断码修复和销量匹配后，无法获得额外补充
2. B级卖场在第三阶段获得大量分配，而SA/A级却无法参与竞争
3. 违背了"高等级卖场优先"的基本原则

#### 解决方案

将第三阶段改为处理所有等级卖场，按等级权重排序：

```python
# 3. 按销尽率优先分配（所有等级参与，按等级权重排序）
all_stores_with_score = []
for store in stores_sorted:
    level = store_data[store]['level']
    sell_through = store_data[store]['sell_through']
    weighted_score = sell_through * level_weights.get(level, 1.0)
    all_stores_with_score.append((store, weighted_score))

# 按综合得分降序排序
all_stores_with_score.sort(key=lambda x: x[1], reverse=True)
```

### 3.2 问题二：第二阶段销量匹配逻辑缺陷

#### 问题定位

**代码位置：** [allocation_core.py:110-128](file:///workspace/allocation_core.py#L110-L128)

```python
# 2. 销量匹配
for store in stores_sorted:
    target_inv = int(store_data[store]['sales_30d'])  # ← 问题：仅用销量作为目标
    to_allocate = min(to_allocate, max_single_size - current_inv)  # ← 问题：上限15件过高
```

#### 问题分析

1. 目标库存仅基于30天销量，未考虑卖场等级差异
2. 上限设置不合理，`max_single_size=15` 件对于销量低的卖场明显过高
3. 未引入安全库存概念

#### 解决方案

引入供应链标准公式计算目标库存：

```
目标库存 = 平均日需求 × 覆盖周期 + 安全库存

其中：
- 平均日需求 = 30天销量 ÷ 30
- 覆盖周期 = 根据卖场等级设定（支持页面配置）
- 安全库存 = 平均日需求 × 安全系数
```

### 3.3 问题三：第三阶段分配上限过高

#### 问题定位

**代码位置：** [allocation_core.py:144-147](file:///workspace/allocation_core.py#L144-L147)

```python
if current_inv < max_single_size:
    to_allocate = max_single_size - current_inv  # ← 问题：直接分配到15件上限
```

#### 问题分析

当前代码将库存直接分配到 `max_single_size=15` 件，不考虑实际销量需求和卖场等级。

#### 解决方案

基于库销比理论设计分配上限公式：

```
分配上限 = max(销量 × 等级权重, 最低保障量)

其中：
- 等级权重 = 根据卖场等级设定（支持页面配置）
- 最低保障量 = 核心尺码2件，非核心尺码1件
```

### 3.4 问题四：等级权重科学评估

#### 评估维度

采用层次分析法（AHP）构建评估模型，评估维度包括：

1. **战略价值**（权重40%）：卖场在品牌战略中的地位
2. **销售贡献**（权重30%）：卖场对整体销售的贡献度
3. **周转效率**（权重30%）：卖场的库存周转速度

#### 等级评分矩阵

| 等级 | 战略价值 | 销售贡献 | 周转效率 | 综合得分 |
|-----|---------|---------|---------|---------|
| SA | 10 | 10 | 9 | 9.7 |
| A | 8 | 8 | 8 | 8.0 |
| B | 6 | 6 | 7 | 6.3 |
| C | 4 | 4 | 6 | 4.6 |
| D | 2 | 2 | 5 | 2.9 |
| OL | 3 | 3 | 6 | 3.9 |

#### 归一化权重计算

```
权重 = 等级得分 ÷ 最高等级得分 × 基准倍数

SA权重 = 9.7 ÷ 9.7 × 1.5 = 1.5
A权重  = 8.0 ÷ 9.7 × 1.5 = 1.3
B权重  = 6.3 ÷ 9.7 × 1.5 = 1.2
C权重  = 4.6 ÷ 9.7 × 1.5 = 0.9
D权重  = 2.9 ÷ 9.7 × 1.5 = 0.6
OL权重 = 3.9 ÷ 9.7 × 1.5 = 0.7
```

### 5.2 各阶段详细说明

#### 阶段1：断码修复

**目的：** 确保每个卖场达到最低库存要求，避免断码

**规则：**
- SA/A级卖场：核心尺码（160/165）至少2件，非核心尺码至少1件
- 其他等级卖场：核心尺码至少1件

**公式：**
```
分配数量 = 目标库存 - 当前库存
```

#### 阶段2：销量匹配

**目的：** 根据销量和卖场等级，分配合理的库存

**公式：**
```
目标库存 = (30天销量 ÷ 30) × 覆盖周期 + (30天销量 ÷ 30) × 覆盖周期 × 安全系数

简化为：
目标库存 = 30天销量 ÷ 30 × 覆盖周期 × (1 + 安全系数)
```

**示例：**
- SA级卖场，30天销量60件：
  - 目标库存 = 60 ÷ 30 × 30 × (1 + 0.5) = 2 × 30 × 1.5 = 90件
- B级卖场，30天销量30件：
  - 目标库存 = 30 ÷ 30 × 14 × (1 + 0.3) = 1 × 14 × 1.3 = 18件

#### 阶段3：销尽率优先分配

**目的：** 优先分配给销尽率高且等级高的卖场

**公式：**
```
综合得分 = 销尽率 × 等级权重
分配上限 = 30天销量 × 等级权重
```

**示例：**
- SA级卖场，销尽率50%：
  - 综合得分 = 0.5 × 1.5 = 0.75
- B级卖场，销尽率70%：
  - 综合得分 = 0.7 × 1.2 = 0.84
- 结果：B级卖场综合得分更高，优先分配

#### 阶段4：剩余分配

**目的：** 将剩余库存按等级优先级分配

**规则：**
- 按等级顺序分配：SA → A → B → C → D → OL
- 同等级内平均分配
- 单卖场上限10件
